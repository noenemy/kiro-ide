# Kiro IDE 스펙 드리븐 개발 실습 시나리오 50선

스펙 드리븐 개발은 바이브코딩과 달리, 요구사항 → 설계 → 태스크 순서로 체계적으로 진행합니다.
각 시나리오는 Kiro의 Spec 워크플로우(Requirements → Design → Tasks)를 활용하여 실습합니다.

---

## 🟢 입문 (Beginner) — 1~2시간

### 시나리오 1: 사용자 프로필 카드 컴포넌트

**예상 소요시간:** 1시간

**프로젝트 개요:**
재사용 가능한 사용자 프로필 카드 UI 컴포넌트를 설계하고 구현한다.

**Spec 시작 프롬프트:**
> "재사용 가능한 사용자 프로필 카드 컴포넌트를 만들고 싶어. 아바타, 이름, 직함, 소셜 링크를 표시하고, 다크/라이트 테마를 지원해야 해."

**요구사항 단계에서 다룰 내용:**
- 프로필 카드에 표시할 데이터 필드 정의 (이름, 직함, 아바타 URL, 소셜 링크 등)
- 테마 지원 범위 (다크/라이트/시스템 설정 따르기)
- 반응형 동작 (모바일/데스크톱)
- 접근성 요구사항 (스크린 리더, 키보드 네비게이션)

**설계 단계에서 다룰 내용:**
- 컴포넌트 Props 인터페이스 설계
- CSS 변수 기반 테마 시스템
- 컴포넌트 상태 관리 (호버, 포커스)
- 폴백 처리 (아바타 없을 때 이니셜 표시)

**태스크 예상 구성:**
1. 프로필 카드 HTML/CSS 기본 구조 작성
2. Props 기반 데이터 바인딩 구현
3. 다크/라이트 테마 전환 구현
4. 반응형 레이아웃 적용
5. 접근성 속성 추가 및 테스트

**실습 포인트:**
- Spec에서 정의한 Props 인터페이스가 실제 구현과 일치하는지 확인
- 요구사항에 없는 기능을 추가하고 싶을 때 Spec을 먼저 업데이트하는 습관 들이기
- 정확성 속성(correctness property): "모든 필수 필드가 렌더링되어야 한다"

### 시나리오 2: 이메일 유효성 검사 라이브러리

**예상 소요시간:** 1시간

**프로젝트 개요:**
다양한 이메일 형식을 검증하는 유틸리티 라이브러리를 만든다.

**Spec 시작 프롬프트:**
> "이메일 주소 유효성 검사 라이브러리를 만들고 싶어. RFC 5322 기본 규칙을 따르고, 일회용 이메일 도메인 차단, MX 레코드 확인 기능도 필요해."

**요구사항 단계에서 다룰 내용:**
- 지원할 이메일 형식 규칙 (로컬 파트, 도메인 파트)
- 유효/무효 이메일 예시 목록
- 일회용 이메일 도메인 블랙리스트 관리 방법
- MX 레코드 확인 옵션 (비동기)
- 에러 메시지 형식

**설계 단계에서 다룰 내용:**
- 함수 시그니처 설계 (`validate(email, options)`)
- 검증 파이프라인 (구문 → 도메인 → MX)
- 에러 타입 열거형
- 캐싱 전략 (MX 레코드 결과)

**태스크 예상 구성:**
1. 기본 구문 검증 함수 구현
2. 도메인 검증 로직 구현
3. 일회용 이메일 도메인 차단 기능
4. MX 레코드 확인 기능 (선택적)
5. Property-based 테스트 작성

**실습 포인트:**
- 정확성 속성: "유효한 이메일은 항상 통과, 명백히 무효한 이메일은 항상 거부"
- 엣지 케이스를 Spec 단계에서 미리 정의하는 것의 가치 체감
- PBT로 랜덤 문자열 입력 시 크래시 없음을 검증

### 시나리오 3: 할 일 관리 REST API

**예상 소요시간:** 1.5시간

**프로젝트 개요:**
할 일(Todo) CRUD REST API를 설계하고 구현한다.

**Spec 시작 프롬프트:**
> "할 일 관리 REST API를 만들고 싶어. 할 일 생성, 조회, 수정, 삭제, 완료 토글 기능이 필요하고, 우선순위와 마감일도 지원해야 해."

**요구사항 단계에서 다룰 내용:**
- API 엔드포인트 목록 (GET/POST/PUT/DELETE)
- 할 일 데이터 모델 (제목, 설명, 우선순위, 마감일, 완료 여부)
- 필터링/정렬/페이지네이션 요구사항
- 에러 응답 형식
- 입력 유효성 검사 규칙

**설계 단계에서 다룰 내용:**
- RESTful 엔드포인트 설계
- 데이터 스키마 (JSON Schema)
- 미들웨어 구조 (유효성 검사, 에러 핸들링)
- 저장소 패턴 (인메모리 → DB 전환 가능)

**태스크 예상 구성:**
1. Express 프로젝트 셋업 + 데이터 모델 정의
2. CRUD 엔드포인트 구현
3. 필터링/정렬/페이지네이션 구현
4. 입력 유효성 검사 미들웨어
5. 에러 핸들링 + 테스트

**실습 포인트:**
- API 설계를 Spec에서 먼저 확정하고 구현하는 흐름 체험
- 정확성 속성: "생성한 할 일은 조회 시 반드시 존재해야 한다"
- Spec 변경 시 영향 범위 파악 연습

### 시나리오 4: 마크다운 → HTML 변환기

**예상 소요시간:** 1.5시간

**프로젝트 개요:**
마크다운 텍스트를 HTML로 변환하는 파서를 직접 구현한다.

**Spec 시작 프롬프트:**
> "마크다운을 HTML로 변환하는 파서를 만들고 싶어. 헤딩, 볼드, 이탤릭, 링크, 코드 블록, 리스트를 지원해야 해. 외부 라이브러리 없이 직접 구현하고 싶어."

**요구사항 단계에서 다룰 내용:**
- 지원할 마크다운 문법 목록과 각각의 HTML 매핑
- 중첩 문법 처리 규칙 (볼드 안의 이탤릭 등)
- XSS 방지 (HTML 이스케이프)
- 지원하지 않는 문법의 처리 방식

**설계 단계에서 다룰 내용:**
- 파싱 전략 (정규식 vs 토크나이저)
- 변환 파이프라인 설계
- 이스케이프 처리 순서
- 확장 가능한 플러그인 구조

**태스크 예상 구성:**
1. 인라인 요소 파싱 (볼드, 이탤릭, 코드, 링크)
2. 블록 요소 파싱 (헤딩, 리스트, 코드 블록)
3. 중첩 문법 처리
4. XSS 방지 이스케이프 처리
5. Property-based 테스트 (임의 마크다운 입력 → 유효한 HTML 출력)

**실습 포인트:**
- 정확성 속성: "변환 결과는 항상 유효한 HTML이어야 한다"
- 파싱 순서가 결과에 미치는 영향을 Spec에서 미리 정의
- PBT로 다양한 마크다운 조합 테스트

### 시나리오 5: 비밀번호 강도 측정기

**예상 소요시간:** 1시간

**프로젝트 개요:**
비밀번호 강도를 분석하고 점수를 매기는 라이브러리를 만든다.

**Spec 시작 프롬프트:**
> "비밀번호 강도 측정 라이브러리를 만들고 싶어. 길이, 문자 다양성, 일반적인 패턴 감지, 사전 단어 포함 여부를 분석해서 0~100 점수와 등급(매우 약함/약함/보통/강함/매우 강함)을 반환해야 해."

**요구사항 단계에서 다룰 내용:**
- 점수 산정 기준 (길이, 대소문자, 숫자, 특수문자, 연속 문자)
- 감점 요소 (일반 패턴: "123456", "password", 키보드 패턴)
- 등급 구간 정의
- 개선 제안 메시지 생성 규칙

**설계 단계에서 다룰 내용:**
- 점수 계산 알고리즘 설계
- 패턴 감지 모듈 구조
- 사전 데이터 관리 방식
- 함수 인터페이스 (`analyze(password) → { score, grade, suggestions }`)

**태스크 예상 구성:**
1. 기본 점수 계산 (길이, 문자 다양성)
2. 패턴 감지 (연속 문자, 반복, 키보드 패턴)
3. 사전 단어 체크
4. 등급 판정 + 개선 제안 생성
5. Property-based 테스트

**실습 포인트:**
- 정확성 속성: "빈 문자열은 항상 점수 0", "20자 이상 + 모든 문자 종류 포함 시 70점 이상"
- 점수 기준을 Spec에서 명확히 정의하는 것의 중요성
- PBT로 점수가 항상 0~100 범위인지 검증

### 시나리오 6: 날짜/시간 유틸리티 라이브러리

**예상 소요시간:** 1.5시간

**프로젝트 개요:**
날짜 포맷팅, 상대 시간 표시, 날짜 계산 기능을 제공하는 유틸리티를 만든다.

**Spec 시작 프롬프트:**
> "날짜/시간 유틸리티 라이브러리를 만들고 싶어. 날짜 포맷팅(YYYY-MM-DD 등), 상대 시간 표시(3분 전, 2일 전), 날짜 간 차이 계산, 영업일 계산 기능이 필요해. 한국어 로케일을 기본 지원해야 해."

**요구사항 단계에서 다룰 내용:**
- 지원할 포맷 패턴 목록
- 상대 시간 표시 규칙 (방금 전, N분 전, N시간 전, N일 전 등)
- 영업일 계산 규칙 (주말, 공휴일)
- 타임존 처리 방침
- 로케일 지원 범위

**설계 단계에서 다룰 내용:**
- 함수별 시그니처 설계
- 로케일 데이터 구조
- 공휴일 데이터 관리 방식
- 불변성 원칙 (원본 Date 객체 변경 금지)

**태스크 예상 구성:**
1. 날짜 포맷팅 함수 구현
2. 상대 시간 표시 함수 구현
3. 날짜 차이 계산 함수
4. 영업일 계산 함수
5. 한국어 로케일 + 테스트

**실습 포인트:**
- 정확성 속성: "포맷 후 파싱하면 원래 날짜와 동일해야 한다" (라운드트립)
- 타임존 엣지 케이스를 Spec에서 미리 정의
- PBT로 임의 날짜에 대한 포맷/파싱 라운드트립 검증

### 시나리오 7: JSON 스키마 유효성 검사기

**예상 소요시간:** 1.5시간

**프로젝트 개요:**
JSON 데이터가 주어진 스키마에 맞는지 검증하는 라이브러리를 만든다.

**Spec 시작 프롬프트:**
> "간단한 JSON 스키마 유효성 검사기를 만들고 싶어. type, required, minLength, maxLength, minimum, maximum, enum, pattern 규칙을 지원하고, 중첩 객체와 배열도 검증할 수 있어야 해."

**요구사항 단계에서 다룰 내용:**
- 지원할 스키마 키워드 목록과 동작 정의
- 중첩 객체/배열 검증 규칙
- 에러 메시지 형식 (경로 + 메시지)
- 다중 에러 수집 (첫 에러에서 멈추지 않음)

**설계 단계에서 다룰 내용:**
- 검증 함수 재귀 구조
- 에러 수집 패턴
- 커스텀 검증 규칙 확장 포인트
- 스키마 자체의 유효성 검사

**태스크 예상 구성:**
1. 기본 타입 검증 (string, number, boolean, null)
2. 문자열/숫자 제약 조건 검증
3. 객체 검증 (required, properties)
4. 배열 검증 (items, minItems, maxItems)
5. 에러 수집 + Property-based 테스트

**실습 포인트:**
- 정확성 속성: "스키마에 맞는 데이터는 항상 통과, 맞지 않는 데이터는 항상 거부"
- 재귀 구조 설계를 Spec에서 미리 계획
- PBT로 임의 JSON + 스키마 조합 테스트

### 시나리오 8: 키-값 저장소 (인메모리 캐시)

**예상 소요시간:** 1시간

**프로젝트 개요:**
TTL(Time-To-Live) 지원 인메모리 키-값 저장소를 만든다.

**Spec 시작 프롬프트:**
> "인메모리 키-값 저장소를 만들고 싶어. get, set, delete, has 기본 연산과 TTL(만료 시간) 지원, 최대 용량 제한(LRU 퇴거), 이벤트 콜백(만료 시, 삭제 시)이 필요해."

**요구사항 단계에서 다룰 내용:**
- 기본 연산 목록과 시그니처
- TTL 동작 규칙 (만료된 키 접근 시 동작)
- LRU 퇴거 정책 상세
- 이벤트 콜백 종류와 타이밍
- 스레드 안전성 (필요 여부)

**설계 단계에서 다룰 내용:**
- 내부 데이터 구조 (Map + 이중 연결 리스트 for LRU)
- TTL 관리 방식 (lazy vs proactive 만료)
- 이벤트 시스템 설계
- 메모리 사용량 추적

**태스크 예상 구성:**
1. 기본 get/set/delete/has 구현
2. TTL 지원 구현
3. LRU 퇴거 정책 구현
4. 이벤트 콜백 시스템
5. Property-based 테스트

**실습 포인트:**
- 정확성 속성: "set 후 TTL 내 get은 항상 값을 반환", "용량 초과 시 가장 오래 미사용된 키가 퇴거"
- LRU 알고리즘을 Spec에서 명확히 정의
- PBT로 임의 연산 시퀀스에 대한 불변 조건 검증

### 시나리오 9: 폼 유효성 검사 프레임워크

**예상 소요시간:** 1.5시간

**프로젝트 개요:**

### 바이브코딩과의 차이점

| 항목 | 바이브코딩 | 스펙 드리븐 |
|------|-----------|------------|
| 시작점 | 프롬프트 한 줄 | 요구사항 문서 |
| 설계 | 암묵적 (AI 판단) | 명시적 (설계 문서) |
| 변경 관리 | 프롬프트 수정 | Spec 업데이트 → 태스크 재생성 |
| 테스트 | 수동 확인 | Property-based 테스트 |
| 적합한 규모 | 소규모/프로토타입 | 중규모 이상/프로덕션 |

### 효과적인 Spec 작성법

1. 요구사항에 "정확성 속성(correctness property)"을 반드시 포함하기
2. 엣지 케이스를 요구사항 단계에서 미리 나열하기
3. 설계 문서에 데이터 흐름 다이어그램 포함하기
4. 태스크는 독립적으로 테스트 가능한 단위로 분리하기
5. Spec 변경 시 영향받는 태스크를 추적하기

### Property-Based Testing 활용

- 모든 시나리오에서 최소 1개 이상의 정확성 속성을 정의
- 속성은 "항상 참이어야 하는 조건"으로 표현
- 예: "정렬 후 배열 길이는 변하지 않는다", "인코딩 후 디코딩하면 원본과 동일하다"
- fast-check (JS) 또는 hypothesis (Python) 라이브러리 활용
